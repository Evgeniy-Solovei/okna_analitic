# Что запросить у заказчика для реализации

Этот чек-лист можно отправлять заказчику почти без изменений. Пароли личных пользователей не нужны: для интеграций нужен отдельный технический пользователь.

## 1. Bitrix24

Нужно:

- адрес портала Bitrix24: `https://...bitrix24.by`;
- отдельный технический пользователь для интеграции, например `analytics.integration`;
- входящий webhook от этого технического пользователя;
- подтверждение, что пользователь имеет права читать CRM.

Права технического пользователя:

- чтение лидов;
- чтение сделок;
- чтение пользователей/ответственных;
- чтение воронок и стадий CRM;
- чтение пользовательских полей CRM;
- чтение истории стадий сделок, если доступно на портале;
- право создать/изменить одно техническое поле сделки для `first_zz_at`, если история стадий окажется недоступной или неполной.

Webhook лучше выпускать не от личного аккаунта директора и не от менеджера. Если у нас будет админ-доступ в Bitrix24, мы можем выпустить webhook сами от технического пользователя.

## 2. ID и поля Bitrix24

Нужно получить или определить вместе с заказчиком:

- ID воронки `B2C Панорама`;
- ID воронки `B2C РО`;
- ID стадии `ЗЗ` в воронке Панорама;
- ID стадии `ЗЗ` в воронке РО;
- ID стадии `ЗН` в обеих воронках;
- поле даты договора;
- поле суммы договора, если сумма не хранится в стандартном `OPPORTUNITY`;
- пользовательское поле лида для направления `Панорама / РО`;
- значения этого поля: как именно записаны `Панорама` и `РО`;
- список менеджеров, которых включать в отчет;
- список пользователей, которым нужен доступ к дашборду.

## 3. Правила, которые нужно подтвердить

Без этих ответов можно начать разработку, но нельзя корректно закрыть приемку:

- По какому ответственному считаем лиды: ответственный лида на дату создания или текущий ответственный?
- По какому ответственному считаем ЦЛ: ответственный сделки на дату создания или текущий ответственный?
- По какому ответственному считаем ЗЗ: ответственный на момент первого входа в ЗЗ или текущий ответственный сделки?
- Что делать, если ответственный сменился после создания сделки?
- При делении на ноль показываем `0`, `0%`, прочерк или `нет данных`?
- Менеджеры видят всю сравнительную таблицу или только себя?
- Нужно ли исключать отдельных пользователей/роботов/админов из менеджерского отчета?

## 4. Контрольные примеры для проверки

Нужны реальные примеры из Bitrix24:

- сделка создана в один день, впервые попала в `ЗЗ` в другой день;
- сделка была в `ЗЗ`, потом ушла дальше, потом вернулась в `ЗЗ`;
- сделка в один день прошла `ЗЗ`, `ЗН` и договор;
- 1-2 контрольных периода, по которым заказчик уже знает правильные цифры;
- ручной отчет/выгрузка из Bitrix24 за эти периоды.

## 5. Сервер

Для production нужен VPS:

- Ubuntu 22.04 или 24.04;
- минимум 2 CPU;
- минимум 4 GB RAM, лучше 8 GB из-за Metabase;
- 50-80 GB SSD;
- SSH-доступ с `sudo`;
- открытые порты `80`, `443`;
- возможность запускать Docker и Docker Compose.

На сервере будет поднято:

- Django;
- PostgreSQL;
- Redis;
- Celery worker;
- Celery beat;
- Metabase;
- Nginx;
- SSL через Let's Encrypt.

## 6. Домен

Нужен домен или поддомен, например:

`analytics.okna-panorama.by`

От заказчика нужно:

- доступ к DNS-зоне домена; или
- чтобы они сами создали `A`-запись на IP сервера.

Также нужен email для SSL-сертификата Let's Encrypt.

## 7. Будущие доступы, не обязательные для первого этапа

Для следующих этапов позже понадобятся:

- Google Ads / Яндекс Директ: доступ к расходам или регулярные CSV-выгрузки;
- 1С: способ интеграции, выгрузка или API;
- список статей расходов и правил финансового учета;
- права на рекламные и финансовые блоки.
